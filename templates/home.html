What is a VC Type (VCT)?

Short answer: it’s the vct value in an SD-JWT VC — a collision-resistant string that identifies the credential’s type (e.g., an “identity_credential”). It tells wallets and verifiers which rules and UI hints apply.

  • Identifier: vct is a case-sensitive, collision-resistant name.
  • Metadata: a VCT can have a JSON object with display[] (multilingual labels & descriptions), claims[] (selection/rendering hints), and an optional schema (JSON Schema 2020-12+).
  • Where it’s found: fetch from the vct URL (HTTPS), from a trusted registry, via ecosystem methods, or from cache; you can pin an integrity hash via vct#integrity.
  • Privacy note: vct is not selectively disclosable — it’s always revealed. Prefer neutral, minimal identifiers.

Based on the IETF SD-JWT VC draft (media type application/dc+sd-jwt and JOSE typ dc+sd-jwt).

Why it matters

The benefits span across the lifecycle of a credential:

  • Interoperability: a wallet can process a credential from any compliant issuer because the vct points to shared type metadata. This eliminates one-off integrations and accelerates adoption.
  • Consistent user experience: multilingual display[] entries ensure that field labels and credential names look the same across issuers and wallets, reducing confusion and translation overhead.
  • Reliable validation: a linked JSON Schema defines required and optional claims. Verifiers can automatically check integrity and issuers can guarantee that credentials follow an agreed structure.
  • Future-proofing: with optional vct#integrity hashes and caching rules, ecosystems can depend on a stable definition over time, even if registries change.
  • Privacy by design: because the vct is always revealed, using short, uninformative identifiers prevents leakage of sensitive information while still enabling metadata lookup.

What you can do

  • Generate VC Types three ways: from an Issuer (OIDC4VCI), from a JSON Schema, or from scratch with assisted AI.
  • Localize once: produce display[] labels and descriptions in up to 30 languages and reuse them everywhere.
  • Publish to a registry with a stable URL and integrity hash, or keep private while you iterate.
  • Discover & reuse existing types from the public registry to avoid duplicates.

Who it's for

  • Project managers who need a shared reference for names, fields, and UI labels.
  • CTOs/architects coordinating issuers, wallets, and verifiers across vendors.
  • Integrators aligning schemas and UX wording without lengthy back-and-forth.

When to use it

  • Before implementation, to agree on the credential type and UI text.
  • When unifying multiple JSON Schemas into a single approved type.
  • When preparing partner documentation and localization.

Standards & interoperability

  • VCT identifier (vct) names the credential type and is a collision-resistant string.
  • Type metadata can include schema (JSON Schema 2020-12+), display metadata (multi-language rendering), and claim metadata (selection/rendering hints).
  • Retrieval: from a URL in vct, from a trusted registry, via ecosystem methods, or cache; optional integrity via vct#integrity.
  • Media type & JOSE: SD-JWT VC uses application/dc+sd-jwt and header typ = dc+sd-jwt (accept both dc+sd-jwt and legacy vc+sd-jwt during transition).

This registry focuses on SD-JWT VC and the VC Type (VCT) metadata used by wallets and verifiers: schema (validation), display[] (multilingual labels/descriptions), and claim hints for selective disclosure.

Learn more in the IETF OAuth Working Group draft: IETF SD-JWT VC Draft.
